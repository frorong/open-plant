`WsiViewer` — Composition API (v1.4.4)

Declarative layer composition API. The monolithic WsiViewerCanvas was removed in v1.4.4. WsiViewer manages the WebGL and overlay canvases, provides ViewerContext, and renders child layer components. See migration guide.

The current implementation uses offscreen WebGL2 accumulation/colorize passes and composites onto the shared overlay canvas. It stays aligned under pan/zoom/rotation/resize and is intended for hotspot overview, not point hit-testing. If WebGL2 initialization fails, the layer logs a warning and skips rendering.

Legacy WsiViewerCanvas removed in v1.4.4+

The monolithic WsiViewerCanvas component and WsiViewerCanvasProps are no longer exported from open-plant as of v1.4.4.

  • Upgrade: use WsiViewer + layer components (see sections above) and the migration guide.
  • Stay on the legacy API temporarily: pin "open-plant": "1.4.3" (or the last version that still shipped WsiViewerCanvas in your lockfile).
  • Historical prop list: preserved in git history and in migration-1.4.0.md mapping tables.
ROI class groups: call computeRoiPointGroups() from app code instead of onRoiPointGroups. customLayers has no direct replacement — use useViewerContext + a host React overlay.

API Stability Policy (WS-10)

SemVer: patch = bug fixes, minor = backward-compatible additions, major = breaking changes.
Deprecation lifecycle: add replacement first, mark deprecated in docs, keep compatibility for at least two minor releases, remove only in the next major.
Migration contract: every breaking change must ship with before/after examples and a checklist.
Known limitations: browser-interaction E2E (real pan/zoom/draw automation) is not yet fully covered in CI.
For upgrade steps and before/after snippets, see Migration Guide.
